Claims Assistant — Project Write-Up

AI-Assisted Insurance Claims Intake Application  |  April 2026

1. Problem & Motivation

Insurance claims intake is tedious and error-prone. Claimants must fill lengthy forms with type-specific fields (auto, property, health), and adjusters must manually review submissions for missing data, logical inconsistencies, and anomalies. I chose this problem because it is a natural fit for LLM-assisted workflows: free-text descriptions contain most of the structured data needed, and validation rules are nuanced enough that rule-based checks alone fall short.

2. How Users Interact

The application centers on a 5-step wizard:

  1. Type & Description — The user selects a claim type (auto, property, health, other) and writes a free-text incident description. Clicking "Extract with AI" sends the text to GPT-4o, which returns structured fields with per-field confidence scores. Only medium/high confidence fields pre-fill the form.
  2. Claimant Info — Name, contact details, and policy number. Pre-filled by AI where possible; users review and correct.
  3. Incident Details — Date, estimated amount, priority, plus type-specific fields (e.g., vehicle info for auto claims, property address for property claims). Also AI-populated.
  4. Documents — File upload for supporting evidence.
  5. Review & Submit — AI validation runs automatically, surfacing errors (blocking) and warnings (overridable). Users see a full summary before submitting.

A separate chat assistant lets internal staff query claims data conversationally — e.g., "show me all urgent property claims from last month" — backed by OpenAI tool calling against four backend operations (search, lookup, stats, recent activity).

3. Key Technical & Product Decisions

  • Monorepo with shared types: Zod schemas in a shared package are the single source of truth for request/response shapes. Both frontend and backend import them, eliminating type drift between layers.
  • Confidence-gated pre-fill: AI extraction returns confidence per field. Low-confidence fields are excluded from auto-fill, reducing the risk of silently inserting wrong data. The UI tracks which fields were AI-filled so users know what to verify.
  • Validation as AI, not rules: Instead of hardcoding validation logic, the system sends the full claim to GPT-4o for completeness and anomaly checks. This catches nuanced issues (incident date in the future, amounts unusual for the claim type) that would require extensive rule maintenance.
  • Graceful degradation: If AI services fail, users can still complete and submit claims manually. Validation failures show a notice but do not block the workflow.
  • Draft-first persistence: The wizard creates a draft claim in the database before document upload, so files attach to a real record. Idempotency checks prevent duplicate drafts.

4. Edge Cases & Limitations

  • Relative date resolution: The current date is injected into the AI prompt so "yesterday" resolves correctly, but ambiguous dates ("last Friday" across month boundaries) can still mis-resolve.
  • Chat iteration cap: The chat assistant limits tool-call iterations to 5 to prevent infinite loops, meaning very complex multi-step queries may get truncated.
  • No authentication: The system has no user auth — all claims are visible to all users and createdBy is a placeholder.
  • No real-time updates: The claims dashboard requires manual refresh; there are no WebSocket or polling mechanisms for status changes.
  • AI cost exposure: Every extraction and validation call hits the OpenAI API with no rate limiting or cost controls beyond max-result caps on search queries.

5. What I Would Improve Next

  1. Authentication & RBAC — Add user roles (claimant, adjuster, admin) so claims are scoped and actions are permissioned.
  2. Streaming chat responses — The chat assistant currently waits for the full response; streaming would improve perceived latency.
  3. Frontend tests — The web app has zero test coverage. Component tests for the wizard steps and integration tests for the AI-assisted flow are overdue.
  4. Cost guardrails — Per-user rate limits on AI calls, plus a fallback to cheaper models for simple extractions.
  5. Claim lifecycle automation — Use AI to suggest next actions (request more info, escalate, approve) based on claim content and history, moving beyond intake into adjudication support.
